# Requirements Document

## Introduction

This feature upgrades the Web Scraper Toolkit's query engine from a rule-based keyword matcher to an AI-powered natural language interface. Instead of mapping tokens to hardcoded CSS selectors, the system will use an LLM (via OpenAI-compatible API) to understand the user's intent from free-form English, generate a precise CSS selector for the scraped page's actual DOM structure, and optionally summarize or interpret the extracted results. The rule-based engine is retained as a fallback when the AI is unavailable or unconfigured.

## Glossary

- **Query Engine**: The backend service that accepts an HTML document and a plain-English intent, and returns matching content.
- **LLM**: Large Language Model — an external AI API (e.g. OpenAI GPT-4o-mini) used to interpret intent and generate selectors.
- **CSS Selector**: A pattern used to select DOM elements in an HTML document.
- **Intent**: A plain-English description of what the user wants to extract from a scraped page.
- **Fallback Engine**: The existing token-scoring rule-based query engine used when the LLM is unavailable.
- **DOM Skeleton**: A compact, truncated representation of the page's HTML structure sent to the LLM to reduce token usage.
- **AI Summary**: A short natural-language explanation of the extracted results, generated by the LLM.
- **API Key**: A secret credential used to authenticate requests to the LLM provider.
- **Groq**: A high-speed LLM inference provider with an OpenAI-compatible API. Used as the AI backend for this project.
- **openai/gpt-4o**: The model served via Groq, used for selector generation and summarization.

## Requirements

### Requirement 1

**User Story:** As a developer using the scraper, I want to type free-form English queries and get accurate results, so that I don't need to know CSS selectors or the page's structure.

#### Acceptance Criteria

1. WHEN a user submits an intent such as "find all article headings" or "get the prices of products", THE Query Engine SHALL send the intent and a DOM skeleton to the LLM and return elements matching the LLM-generated CSS selector.
2. WHEN the LLM returns a CSS selector, THE Query Engine SHALL apply that selector to the full HTML and return all matching text, href, or src values.
3. WHEN the LLM produces a selector that matches zero elements, THE Query Engine SHALL fall back to the rule-based engine and include a `fallback: true` flag in the response.
4. WHEN the LLM API is unreachable or returns an error, THE Query Engine SHALL fall back to the rule-based engine without exposing internal error details to the client.

---

### Requirement 2

**User Story:** As a developer, I want the AI to summarize what it found after extraction, so that I can understand the results at a glance without reading every item.

#### Acceptance Criteria

1. WHEN extraction returns one or more results, THE Query Engine SHALL include a short AI-generated `summary` field in the response describing what was found.
2. WHEN extraction returns zero results, THE Query Engine SHALL include a `summary` explaining that nothing matched and suggesting a rephrased query.
3. WHILE the summary is being generated, THE system SHALL return the extracted results immediately and stream or append the summary asynchronously so the user is not blocked.

---

### Requirement 3

**User Story:** As a system administrator, I want to configure the AI provider and model via environment variables, so that I can switch providers or disable AI without changing code.

#### Acceptance Criteria

1. THE system SHALL read the LLM provider API key from an environment variable named `GROQ_API_KEY`.
2. THE system SHALL read the model name from an environment variable named `AI_MODEL`, defaulting to `openai/gpt-4o` when not set.
3. THE system SHALL use Groq's OpenAI-compatible base URL (`https://api.groq.com/openai/v1`) as the default, overridable via `AI_BASE_URL`.
4. WHEN `GROQ_API_KEY` is not set or is empty, THE system SHALL skip the LLM call entirely and use the rule-based fallback engine.

---

### Requirement 4

**User Story:** As a developer, I want the AI to only receive a compact skeleton of the page structure rather than the full HTML, so that token costs stay low and latency is acceptable.

#### Acceptance Criteria

1. WHEN preparing a prompt for the LLM, THE Query Engine SHALL send only a DOM skeleton containing tag names, id and class attributes, truncated to a maximum of 4000 characters.
2. THE system SHALL strip all script, style, and inline event attributes before building the skeleton.
3. WHEN the full page skeleton exceeds 4000 characters, THE system SHALL truncate it with a trailing `...` marker so the LLM is aware the document continues.

---

### Requirement 5

**User Story:** As a user, I want the frontend to clearly show whether a result came from AI or the fallback engine, so that I understand the confidence level of the extraction.

#### Acceptance Criteria

1. WHEN the query response includes `ai_powered: true`, THE frontend SHALL display an "AI" badge next to the results count.
2. WHEN the query response includes `fallback: true`, THE frontend SHALL display a "Rule-based" badge and a note that AI was unavailable.
3. WHEN an AI summary is present in the response, THE frontend SHALL render it in a distinct summary block above the results list.
